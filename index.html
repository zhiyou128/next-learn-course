What is Next.js?

The folder-based system that turns your project structure into a working web app

The most magical thing about the web

Every time you type a URL and press Enter, something extraordinary happens in under a second. But have you ever wondered what's actually happening between that keystroke and the page appearing?

💡
The "behind-the-scenes" moment

You've used web apps your whole life. Now you're going to see exactly how they work — because that's how you learn to direct AI to build them for you.

A framework called Next.js sits between "user wants to see something" and "user sees it." It's the director of a very busy, very fast play.

Four actors, one page load

When you visit a URL, four characters talk it out behind the scenes. Watch their conversation — it's the core of how every web app works.

U
User (You)

I want to see the dashboard — I typed the URL into my browser.

B
Browser

Got it — I'll ask the Next.js Server for the page at this URL.

N
Next.js Server

URL received. The folder structure says app/dashboard/page.tsx handles this route. Let me find that component.

D
Database

Here's the data for this user's dashboard — their name, recent activity, and settings.

N
Next.js Server

Data received. I'll combine it with the page component and send back the finished HTML.

B
Browser

HTML received! I'm rendering the page now so you can see it on screen.

?

The key insight: Next.js is the middleman that figures out what to show based on where your files are, not on complex configuration files.

The folder is the map

Here's the secret that makes Next.js special: you don't write complex routing logic. Instead, the App Router uses your folder names as the address map. Folder = URL. File = page.

app/ The main pages folder — your folder structure becomes the URL map
dashboard/ Becomes /dashboard in the URL
page.tsx The component that renders the /dashboard page
login/ Becomes /login in the URL
page.tsx The login page component
layout.tsx The shared shell that wraps every page (nav, footer)
page.tsx The homepage — renders at /
components/ Reusable UI building blocks shared across pages
lib/ Shared logic and utility functions
ℹ️
Why this matters for working with AI

When you tell an AI "add a Contact page," you need to know it goes in app/contact/page.tsx. Understanding the app/ folder structure is the first step to directing AI effectively.

服务器 vs 客户端 — 代码在哪里运行？

有些代码在后厨（服务器）运行，有些在你桌边（浏览器）运行

每个应用都有两个用餐区

🏠
前台 — 浏览器（客户端）

用户坐的地方。他们看菜单、点击按钮、在表单中输入。快速且交互性强——但他们无法访问数据库或密钥。

👨‍🍳
后台 — 服务器

食物制作的地方。可以访问一切——数据库、API、密钥。但离食客很远，所以对点击的响应较慢。

💡
关键问题

当你点击一个按钮时，决策是在你的桌边（浏览器）还是厨房（服务器）做出的？把代码放错地方会悄悄破坏功能。

服务器组件是默认选项

默认情况下，Next.js 中的每个服务器组件都在服务器上运行。不会为其发送任何客户端 JavaScript。

📊

显示数据

从数据库获取数据、读取文件、调用 API——在服务器上做这些都很安全，密钥不会暴露。

🎨

纯 UI

静态布局、字体排版、设计系统——任何不需要用户交互的渲染。

🔒

安全敏感

任何使用 API 密钥、密码或不应该离开服务器的业务逻辑。

🗄️
须知

你可以通过没有特殊标记来识别服务器组件——因为它是默认的。不需要指令。

当你需要交互性时：'use client'

有些组件需要响应点击、跟踪输入或与浏览器 API 通信。这时你需要添加 'use client' 指令。

CODE 
'use client';
// This file runs in the BROWSER, not on the server

import { lusitana } from '@/app/ui/fonts';
import { AtSymbolIcon } from '@heroicons/react/24/outline';

export default function LoginForm() {
  // useState and event handlers live HERE
  return (
    <form className="space-y-3">
      {/* form fields with onClick, onChange */}
    </form>
  );
}
PLAIN ENGLISH

'use client' 告诉 Next.js：「把这代码发送到浏览器。」

注释是给你的：让你一眼就知道这个文件在哪里运行。

导入字体：和服务器组件版本一样——这里没有区别。

函数是一样的——但现在它可以使用 useState、onClick 处理程序等。

JSX 标记：表单本身看起来一样。只有能力不同。

💡
关键洞察

默认情况下，Next.js App Router 中的所有组件都是服务器组件。只有当你需要交互性——点击处理程序、状态（useState）或浏览器 API 时，才添加 'use client'。

什么真正发送到浏览器？

不是所有代码都会到达用户。以下是服务器组件保留给自己的内容。

// This stays on the server — NEVER reaches browser
const dbPassword = "super-secret-key-123";
await fetchRevenue();

// Database queries
const customers = await db.query("SELECT * FROM users");
// HTML and CSS arrive — the rendered result
<div class="revenue-chart">
  <h2>Monthly Revenue</h2>
  <svg>...chart...</svg>
</div>

/* CSS styles */
// ONLY LoginForm's JS bundle ships
'use client';
export default function LoginForm() {
  const [state, setState] = useState();
  return <form>...</form>;
}

服务器组件将密钥、数据库和重型逻辑保留在服务器上。浏览器只收到 HTML、CSS 和交互式组件的 JavaScript。

服务器 vs 客户端：谁来处理点击？

观察当用户点击按钮时会发生什么——每个组件都认为应该由另一个来处理。

S
Server Component

我准备好了数据。按钮应该只显示我发送的内容。

C
Client Component

没那么快。点击事件发生在浏览器里——那是我的领域。

S
Server Component

没错——但我可以在页面加载前预取数据。然后你只管显示就行。

C
Client Component

成交。你做重活（数据库）。我处理有趣的部分（点击、悬停、表单）。

?

这种分工很自然：服务器组件获取并准备数据。客户端组件处理事件处理程序和状态。

数据获取 — 仪表板如何获取数据

数据库查询、Promise.all 并行请求和 SQL 模板字面量

Screen 1 of 6
📊
你看到的：图表、数字、表格

收入指标、发票列表、客户数量 — 全在一个页面上

❓
但这一切都是从哪里来的？

没有 API 路由，没有 fetch() 调用 — 数据就这样...出现了

🗄️
答案：数据库

服务器组件访问数据库，精确获取每个组件需要的数据

在 Next.js App Router 中，组件可以直接与数据库通信。不需要 API 层 — 组件本身就是 API 消费者。

💡
数据库是一个图书馆员

它不思考。它不决定。它只是检索你要求的内容。lib/data.ts 中的 fetch 函数就像图书馆员的请求：「给我所有状态为『已支付』的发票，按日期排序。」

📖

图书馆员的目录

数据库以表格的形式存储数据。每个表格就像一个目录抽屉：发票、客户、收入。

🔍

完美的记忆力

用 SQL 向数据库提问，它会立即检索答案。它从不忘记，从不犯错。

⚡

无需额外步骤

在 App Router 中，你可以直接在服务器组件内部 await 数据库调用。中间不需要 API 路由。

这里的数据库是 PostgreSQL，一种以有组织的表格存储数据并可以使用 SQL 查询的数据库。

并行查询：同时开始所有问题

想象一下，同时向图书馆员提出三个问题，而不是等待每个答案后再问下一个。这就是 Promise.all。

CODE 

// All three queries start at the same time
const invoiceCountPromise = sql`SELECT COUNT(*) FROM invoices`;
const customerCountPromise = sql`SELECT COUNT(*) FROM customers`;
const invoiceStatusPromise = sql`SELECT SUM(CASE WHEN status = 'paid' THEN amount ELSE 0 END)...`;

// Wait for all three to finish — in parallel
const data = await Promise.all([
  invoiceCountPromise,
  customerCountPromise,
  invoiceStatusPromise,
]);
PLAIN ENGLISH

让图书馆员计算所有发票——但先不要等待答案。

让图书馆员计算所有客户——同时开始。

让图书馆员计算已支付与待处理的总额——也是同时开始。

现在等待所有三个答案同时返回。

Promise.all 将三个进行中的请求捆绑在一起，只有当三个都有答案时才解析。

所有三个都在任何一个完成之前开始——这就是并行获取。

1

三个查询同时触发

→
2

数据库同时处理它们

→
3

所有结果同时返回

SQL 模板字面量

向数据库提问的更安全方式——它可以防止黑客注入恶意代码。模板字面量语法告诉数据库什么是问题，什么是值。

⚡

并行获取

不是问一个问题、等待、然后问下一个——用 Promise.all 同时开始所有问题。总等待时间变成最慢的那个，而不是三个的总和。

1
用户点击链接或刷新 /dashboard

2

3

这部分不需要客户端 JavaScript——服务器发送了完整的 HTML

Server Actions

How users create, update, and delete data

Fetching data is half the story

The other half? When a user creates an invoice, edits a customer, or deletes a record. In Next.js, Server Actions handle all of it.

📥
You already know the read path

Data flows from the database up to the page — that's data fetching. Every page you've built so far.

📤
Server Actions add the write path

Data flows from the page down to the database — creating, updating, or deleting records. The same portal, reversed.

💡
Why this matters with AI

AI coding agents often write separate API routes + fetch calls for mutations. In Next.js App Router, Server Actions are the idiomatic pattern. Knowing this helps you guide AI toward the right architecture.

The suggestion box metaphor

Imagine a restaurant with a suggestion box. A customer fills out a card (the form submission), drops it in, and walks away. Behind the scenes, staff retrieve the card, read it, and update the reservation book. The customer never watches the kitchen work — they just get a result.

1
Customer fills out the card

User fills in the invoice form — customer name, amount, status, date.

2
Card goes into the suggestion box

Form submission sends the data to a Server Action behind the scenes.

3
Staff reads and validates it

Server Action checks: is the amount actually a number? Did they select a customer? Validation catches bad data.

4
Staff updates the reservation book

Validated data gets written to the database — a new invoice record is created.

5
Customer gets redirected to the dining room

After writing, the server tells the browser to go to the invoices list — fresh data loaded.

Inside a Server Action

This is the Server Action that creates invoices. It validates, writes, and redirects — the full journey of a form submission.

CODE 

'use server';

import { z } from 'zod';
import postgres from 'postgres';
import { revalidatePath } from 'next/cache';
import { redirect } from 'next/navigation';

const sql = postgres(process.env.POSTGRES_URL!, { ssl: 'require' });

const FormSchema = z.object({
  customerId: z.string({ invalid_type_error: 'Please select a customer.' }),
  amount: z.coerce.number().gt(0, { message: 'Please enter an amount greater than $0.' }),
  status: z.enum(['pending', 'paid'], { invalid_type_error: 'Please select an invoice status.' }),
});

export async function createInvoice(prevState, formData) {
  // 1. Validate form fields using Zod
  const validatedFields = FormSchema.safeParse({
    customerId: formData.get('customerId'),
    amount:     formData.get('amount'),
    status:     formData.get('status'),
  });

  if (!validatedFields.success) {
    return { errors: validatedFields.error.flatten().fieldErrors,
             message: 'Missing Fields. Failed to Create Invoice.' };
  }

  // 2. Prepare data for the database
  const { customerId, amount, status } = validatedFields.data;
  const amountInCents = amount * 100;
  const date = new Date().toISOString().split('T')[0];

  // 3. Insert into database
  try {
    await sql`INSERT INTO invoices
      (customer_id, amount, status, date)
      VALUES (${customerId}, ${amountInCents}, ${status}, ${date})`;
  } catch (error) {
    return { message: 'Database Error: Failed to Create Invoice.' };
  }

  // 4. Revalidate cache and redirect
  revalidatePath('/dashboard/invoices');
  redirect('/dashboard/invoices');
}

The main function — called when the form is submitted. It receives the previous state and the raw form data from the submitted form.

Ask Zod to check each form field against our rules — is customerId present? Is amount actually a number?

If validation fails, stop here and return error messages. The form will display these errors to the user so they can fix them and try again.

'use server' — file-wide vs inline

You can put 'use server' at the top of an entire file, or inline inside a function. Here's the difference.

📄

'use server' at file top

The entire file runs on the server. Every function in the file — even helper functions called indirectly — runs server-side. This is the pattern used by app/lib/actions.ts.

⚡

'use server' inline

Only the immediately following function runs on the server. The rest of the file can run on the client. Useful when you want to mix client interactivity with server operations.

💡
The whole file, not just one function

When you put 'use server' at the top of a file, it means the entire file runs on the server. Any imports that file pulls in (like the Postgres driver or Zod) also run server-side. This is why app/lib/actions.ts can directly connect to the database — the whole file is a server-only environment.

Why revalidatePath?

After writing to the database, the code calls revalidatePath. Here's why that matters.

🧠
Next.js remembers pages

When you visit a page, Next.js caches it. The next visitor gets the saved version instead of hitting the database again.

🗑️
revalidatePath says "forget what you remembered"

After a new invoice is created, the cached invoices page is now stale. Calling revalidatePath wipes that cache so the next request gets fresh data from the database.

🔄
Then redirect sends the user there

The browser navigates to /dashboard/invoices, which now fetches the updated data — including the new invoice — fresh from the database.

ℹ️
Performance without the complexity

In a traditional app, you'd need to manually refetch data after a mutation. revalidatePath + redirect handles this automatically. The user sees updated data without a page refresh flicker.

React components that run only on the server. Fetch data, query the database, return HTML. Default in App Router.

Functions marked with 'use server' that run on the server and handle form submissions and database writes.

Raw SQL queries written as tagged template literals — the simplest way to talk to a Postgres database.

A Next.js function that clears the cache for a route after a database write — so the next visit shows fresh data.
