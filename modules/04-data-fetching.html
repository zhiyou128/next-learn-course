数据获取 — 仪表板如何获取数据

数据库查询、Promise.all 并行请求和 SQL 模板字面量

Screen 1 of 6
📊
你看到的：图表、数字、表格

收入指标、发票列表、客户数量 — 全在一个页面上

❓
但这一切都是从哪里来的？

没有 API 路由，没有 fetch() 调用 — 数据就这样...出现了

🗄️
答案：数据库

服务器组件访问数据库，精确获取每个组件需要的数据

在 Next.js App Router 中，组件可以直接与数据库通信。不需要 API 层 — 组件本身就是 API 消费者。

💡
数据库是一个图书馆员

它不思考。它不决定。它只是检索你要求的内容。lib/data.ts 中的 fetch 函数就像图书馆员的请求：「给我所有状态为『已支付』的发票，按日期排序。」

📖

图书馆员的目录

数据库以表格的形式存储数据。每个表格就像一个目录抽屉：发票、客户、收入。

🔍

完美的记忆力

用 SQL 向数据库提问，它会立即检索答案。它从不忘记，从不犯错。

⚡

无需额外步骤

在 App Router 中，你可以直接在服务器组件内部 await 数据库调用。中间不需要 API 路由。

这里的数据库是 PostgreSQL，一种以有组织的表格存储数据并可以使用 SQL 查询的数据库。

并行查询：同时开始所有问题

想象一下，同时向图书馆员提出三个问题，而不是等待每个答案后再问下一个。这就是 Promise.all。

CODE 

// All three queries start at the same time
const invoiceCountPromise = sql`SELECT COUNT(*) FROM invoices`;
const customerCountPromise = sql`SELECT COUNT(*) FROM customers`;
const invoiceStatusPromise = sql`SELECT SUM(CASE WHEN status = 'paid' THEN amount ELSE 0 END)...`;

// Wait for all three to finish — in parallel
const data = await Promise.all([
  invoiceCountPromise,
  customerCountPromise,
  invoiceStatusPromise,
]);
PLAIN ENGLISH

让图书馆员计算所有发票——但先不要等待答案。

让图书馆员计算所有客户——同时开始。

让图书馆员计算已支付与待处理的总额——也是同时开始。

现在等待所有三个答案同时返回。

Promise.all 将三个进行中的请求捆绑在一起，只有当三个都有答案时才解析。

所有三个都在任何一个完成之前开始——这就是并行获取。

1

三个查询同时触发

→
2

→
3

所有结果同时返回

你会到处看到的三个模式

🔗

SQL 模板字面量

向数据库提问的更安全方式——它可以防止黑客注入恶意代码。模板字面量语法告诉数据库什么是问题，什么是值。

⚡

并行获取

不是问一个问题、等待、然后问下一个——用 Promise.all 同时开始所有问题。总等待时间变成最慢的那个，而不是三个的总和。

🛡️

参数化查询

当你在搜索「发票 #123」时，搜索词与查询分开发送——就像给图书馆员一张密封信封里的纸条。这可以防止 SQL 注入。

注意 fetchFilteredInvoices 如何将 ILIKE 与参数一起使用：${`%${query}%`}。% 符号意味着「包含」——所以搜索「inv」会找到「invoice」、「invitation」等。

从点击到内容：完整旅程

1
用户点击链接或刷新 /dashboard

浏览器向服务器请求仪表板页面

2
服务器组件调用一个 fetch 函数

在组件内部调用 fetchRevenue() 或 fetchCardData()——就像向图书馆员下订单

3
函数向 PostgreSQL 发送 SQL 查询

使用标记模板字面量：sql`SELECT * FROM revenue`——数据库以它能理解的格式接收问题

4
数据库搜索并返回行

数据库不思考——它只是检索匹配的行并发送回去

5
组件接收数据并渲染 HTML

await 关键字在数据到达后解除阻塞，组件从结果构建 HTML

6
浏览器显示完全渲染的页面

这部分不需要客户端 JavaScript——服务器发送了完整的 HTML

ℹ️
为什么不需要 API 路由？

在传统的网络应用中，你需要像 /api/revenue 这样的 API 端点。App Router 让服务器组件跳过这一步——它们直接与数据库对话，就像图书馆员无需中间人就能伸手到书架上取书。
